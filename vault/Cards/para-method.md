# PARA 方法

PARA 是 Tiago Forte 在「Building a Second Brain」體系中提出的數位資訊組織框架。它的核心問題不是「這個資訊屬於什麼主題？」而是「這個資訊離行動有多近？」——用可行動性（actionability）作為分類的第一原則，將所有數位資訊分配到四個層級。這個設計讓資訊系統自動對齊你當下的優先事項，而不是成為一個永遠在整理卻用不到的知識博物館。

## 核心機制

PARA 的四個層級是一個從「最靠近行動」到「最遠離行動」的光譜：

**Projects（專案）**：有明確目標和截止日的短期努力。例如「完成 Q2 產品提案」「搬家」「準備演講」。專案是唯一會「完成」的層級——完成後它的素材要不歸入 Areas，要不歸入 Archives。判斷標準：能在一句話裡說出「做完的樣子」→ 這是 Project。

**Areas（領域）**：你持續維護、沒有終點的責任範圍。例如「健康」「財務」「團隊管理」「產品設計」。Areas 不會完成，只會演化。與 Projects 的關鍵區別：Project 有 deadline，Area 沒有。很多人把 Area 誤設為 Project（如「減肥」），導致永遠完不成的挫敗感。

**Resources（資源）**：你感興趣但不直接對應任何責任的主題參考資料。例如「機器學習」「咖啡沖煮」「日本建築」。Resources 是低壓力的知識儲備——你不需要對它負責，只是覺得有趣或未來可能用到。

**Archives（封存）**：前三個層級中不再活躍的項目。已完成的 Project、不再維護的 Area、失去興趣的 Resource，全部移到 Archives。不是刪除——是讓它們不再佔用你的注意力。Archives 的存在讓你放心地把東西「收起來」，而不是糾結要不要刪掉。

**層級之間的流動**：PARA 不是靜態分類，而是動態系統。一個 Resource（「學習 Rust」）可能因為新專案需要而升格為 Project（「用 Rust 重寫後端服務」）。一個 Project 完成後，其中的通用知識可能沉澱為 Resource。這種流動性是 PARA 最常被忽略但最重要的特性。

## 由來與故事

Tiago Forte 是一位生產力教練和課程設計師，他在 2017 年開始系統化教授「Building a Second Brain」（BASB）線上課程。PARA 是 BASB 體系中的組織框架，但它的起源早於課程本身。

Forte 曾在一次訪談中提到，他最初的痛點是作為自由工作者同時管理多個客戶專案——他嘗試過按主題分類（資料夾爆炸）、按時間排列（找不到東西）、按工具分類（資訊孤島）。最終他意識到，所有分類系統都在回答錯誤的問題。真正的問題不是「這個東西是什麼」，而是「我現在需要拿它來做什麼」。

PARA 的四層結構直接來自 GTD（Getting Things Done）的「行動層級」概念，但 Forte 把它從任務管理擴展到了所有數位資訊——筆記、文件、書籤、截圖、靈感片段，全部用同一套框架組織。2022 年他出版《Building a Second Brain》一書，PARA 從此成為數位筆記圈最廣泛採用的組織框架之一。

## 案例

**自由工作者的多專案管理**：一位 UX 設計師同時接三個客戶專案。她在 Projects 下建立三個資料夾，每個包含該客戶的合約、設計稿、會議紀錄、反饋文件。當某個專案結案，整個資料夾移入 Archives——不用整理，不用拆分，只是移動。下次接到類似客戶時，她從 Archives 拉出舊專案參考，幾分鐘就能找到當時的設計決策和客戶偏好。

**研究生的論文寫作**：一位博士生用 PARA 管理論文研究。Areas 放「方法論」「導師關係」「學術寫作」等持續維護的領域；Projects 放「Chapter 3 初稿」「期刊投稿」等有截止日的任務；Resources 放感興趣但不直接相關的延伸閱讀。關鍵轉變：他不再把所有論文相關資料丟進一個巨大的「論文」資料夾，而是按行動性拆分——要立刻處理的（Project）、要持續注意的（Area）、有空再看的（Resource）。

**團隊知識庫的結構設計**：一個 10 人的產品團隊用 PARA 重組 Notion workspace。Projects 對應每個 sprint 和跨 sprint 的 initiative；Areas 對應團隊持續關注的面向（用戶研究、技術債、onboarding）；Resources 放競品分析、設計規範、技術文件；Archives 收已完成的 sprint 和廢棄的方案。新成員加入時，看 Projects 知道「現在在做什麼」，看 Areas 知道「團隊在乎什麼」。

## 相關名詞

### 抽象層（往上）
這個概念屬於哪個更大的框架或理論？

- **個人知識管理（PKM）** — PARA 是 PKM 領域中最主流的組織框架，定位於「結構層」，解決「東西放哪裡」的問題
- **Building a Second Brain（BASB）** — PARA 是 BASB 方法論的四大支柱之一（Capture / Organize / Distill / Express），負責 Organize 環節
- **資訊架構（Information Architecture）** — PARA 可以理解為個人尺度的資訊架構設計，用行動性取代傳統的主題分類法

### 橫向探索（同層）
有哪些兄弟概念，跟這個概念在同一個鄰近空間？

- **[[zettelkasten|卡片盒筆記法]]** — Zettelkasten 以原子筆記和連結為核心，PARA 以行動性分層為核心，兩者在 PKM 領域互補：PARA 管「放哪裡」，Zettelkasten 管「怎麼連」
- **GTD（Getting Things Done）** — David Allen 的任務管理系統，PARA 的行動性分層直接受 GTD 影響，但將範圍從任務擴展到所有數位資訊
- **Johnny Decimal** — 另一種數位檔案組織法，用嚴格的數字編號系統替代主題資料夾，與 PARA 的行動性導向形成對比
- **ACCESS 系統** — Nick Milo 提出的知識管理框架（Atlas / Calendar / Cards / Extra / Sources / Spaces），比 PARA 更偏向知識連結而非行動管理

### 拆解層（往下）
這個概念由哪些子元件或具體實作組成？

- **Projects 清單** — PARA 的第一層，所有有明確終點的短期努力，是整個系統的行動核心
- **Areas of Responsibility** — 持續維護的責任範圍，與 Projects 的區別（有/無 deadline）是 PARA 最重要的判斷
- **Progressive Summarization** — Forte 的配套方法，用分層高亮在 Resources 中逐步萃取重點，讓資源在需要時能快速啟用
- **PARA 遷移（Migration）** — 定期檢視並移動項目到正確層級的維護動作，防止系統僵化

## 應用方式

**啟動 PARA 的最小可行步驟：** 不需要把現有資料全部重新分類。從此刻開始，每個新進入系統的資訊只問一個問題：「這跟我正在推進的哪個 Project 有關？」有 → 放進該 Project；沒有但跟持續責任有關 → Area；只是感興趣 → Resource。舊資料？等到你需要用到它的那天再歸位。

**Project 與 Area 的判斷公式：** 如果你能回答「做完長什麼樣？」→ 它是 Project。如果你的回答是「嗯⋯⋯沒有做完的一天」→ 它是 Area。常見陷阱：「學英文」是 Area（沒有終點），「考過 TOEFL 100 分」是 Project（有明確結束條件）。

**跨工具一致性：** PARA 的力量在於所有工具用同一套結構。你的 Notion、Google Drive、桌面資料夾、甚至 email 標籤，都可以用 P-A-R-A 四層來組織。當你需要某個資訊時，不用記「我存在哪個工具裡」，只需要判斷「它屬於哪個層級」。

**定期清理的節奏：** 每週檢視 Projects（有沒有新的？有沒有完成的？），每月檢視 Areas（責任範圍有變化嗎？），每季把不再活躍的項目批量移入 Archives。不做這個清理，系統會在三個月內退化為一個分類稍好的垃圾場。

## 常見偏誤 vs 反直覺

### 常見偏誤

- **主題分類的慣性** — 最常見的錯誤是把 PARA 當成主題資料夾的翻版，建出「PARA > Resources > 心理學 > 認知偏誤 > 確認偏誤」這種深層結構。PARA 的重點是扁平化——每個層級下直接放項目，不需要子分類。深層結構意味著你還在用主題思維

- **Area 膨脹症** — 把太多東西設為 Area，導致持續維護的壓力過大。典型症狀：你的 Areas 有 20 個以上。解法是誠實問自己「我真的在持續維護這個嗎？」如果答案是否，它應該是 Resource 或 Archives

- **不遷移的靜態系統** — 設定好 PARA 後就不再動它，三個月後 Projects 裡全是已完成的專案，Archives 是空的。PARA 必須流動——它是一個動態分配系統，不是一次性的分類行為

### 反直覺洞見

- **少分類反而找得到** — 直覺告訴我們越精細的分類越容易找到東西，但研究顯示人在 3–5 個類別中搜索的效率遠高於 20 個以上。PARA 只有四個頂層類別，這不是簡化——這是認知科學支持的設計決策

- **完成的專案不該被整理，應該被封存** — 很多人在專案結束後花大量時間「把有用的東西提取出來歸檔到對應主題」。PARA 的建議是：直接把整個 Project 資料夾移入 Archives，保持原樣。未來需要時按關鍵字搜尋即可。花時間整理的投資報酬率極低，因為你很可能永遠不會再用到那些資料

- **你的系統反映你的生活，不是反過來** — PARA 結構會隨著你的人生階段改變。轉職後 Projects 和 Areas 會大幅重組，這不是「系統壞了」，而是系統在正常運作。如果你的 PARA 一年沒有大變動，更可能的解釋是你沒在用它，而不是你的生活穩定
